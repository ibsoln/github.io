Logging

    +

    Description — Introducing Couchbase Sync Gateway’s logging functionality and how-to exploit its different options.
    Abstract — Sync Gateway’s _Continuous Logging_ feature delivers flexible log generation and retention, without compromising the availability of diagnostic information necessary to provide effective support and maintenance.

    Overview

    Sync Gateway logs are written to separate files by log level, with each log level supporting individual retention policies. You can control what is logged using configuration file settings and-or the REST API.

    Continuous logging

    [1] provides flexibility in how logs are generated and retained, whilst maintaining the level of logging required by Couchbase Support for investigation of issues.

    Console logging

    can be configured independently, providing additional flexibility for system administrators depending on their needs.
    This allows system administrators running Sync Gateway to tweak the log level, and log keys for the console output to suit their needs, whilst maintaining the level of logging required by Couchbase Support for investigation of issues.

    Configuration

    You configure your logging requirements in the sync-gateway-config.json file, using the logging properties — see: Configuration File and Example 1.

    Example 1. Sample Logging Configuration
    {
  "logging": {
    "log_file_path": "/var/tmp/sglogs",
    "console": {
      "log_level": "debug",
      "log_keys": ["*"]
      },
    "error": {
      "enabled": true,
      "rotation": {
        "max_size": 20,
        "max_age": 180
        }
      },
    "warn": {
      "enabled": true,
      "rotation": {
        "max_size": 20,
        "max_age": 90
        }
      },
    "info": {
      "enabled": false
    },
    "debug": {
      "enabled": false
      }
    },
  "databases": {
    "db": {
      "server": "couchbase://localhost",
      "username": "username",
      "password": "password",
      "bucket": "default",
      "users": {"GUEST": {"disabled": false,"admin_channels": ["*"]}},
      "allow_conflicts": false,
      "revs_limit": 20
      }
    }
  }

    Console Logging

    You can configure console logging independently, providing additional flexibility, without impacting the continuous logging output. For example, System administrators running Sync Gateway at the console may:

    • tweak Log Levels and Log Keys to suit their needs

    • filter console output by log levels and log keys

    • set a color for log output based on log level

    The console output of Sync Gateway can be filtered down via Log Levels and Log Keys, and you can tweak this as much as you like without impacting Support’s ability to analyze the log files.

    Log Levels

    Sync Gateway logs at one (or more) of six defined log levels: none; error; warn; info; debug and trace.

    You can set individual retention policies for each log level, with logs for each level being written to separate files.

    Set log levels in your configuration file — see the logging.$level reference.

    Log levels are additive, so if you enable info level, then warn and error logs are also enabled.

    The default log level is info.

    Log Level Appearance Description

    none

    -

    Disables log output

    error

    [ERR]

    Displays errors that need urgent attention

    warn

    [WRN]

    Displays warnings that need some attention

    info

    [INF]

    Displays information about normal operations that don’t need attention

    debug

    [DBG]

    Displays verbose output that might be useful when debugging

    trace

    [TRC]

    Displays extremely verbose output that might be useful when debugging

    Log Keys

    Log keys provide fine-grained control over the information to be logged. By default, only HTTP is enabled, other available keys include: Auth; Bucket; CRUD; DCP and Import.

    See the logging.console.log_level property reference for a comprehensive list of available keys.

    Set Log Color

    You may set a color for log output based on log level by using logging.console.color_enabled is set to true

    This setting is always disabled on Windows for compatibility reasons.

    Redirect Console Log

    You can easily redirect the console log output to a file. This can be useful when you have specialized logging requirements, such as centralized logging. Just redirect the output and then apply your own log collection mechanism to feed that data elsewhere — see Example 2.

    Example 2. Console Log Redirection
    # Start Sync Gateway and redirect console output to a file
./sync-gateway > my_sg_logs.txt 2>&1

# Start log collection to send to a centralized log aggregator.
logcollector my_sg_logs.txt

    Log File Outputs

    Sync Gateway produces four separate log files, split by log level. Each log file has its own guaranteed retention period.

    The log files can be collected with SGCollect Info for analysis by Couchbase Support when diagnosing Sync Gateway issues.

    Log files are intended solely for the use of Couchbase Support. If you require special log handling then use Redirect Console Log.
    Table 1. Available Log Files
    Log File Default enabled Default max_age Minimum max_age

    sg_error.log

    true

    360 Days

    180 Days

    sg_warn.log

    true

    180 Days

    90 Days

    sg_info.log

    true

    6 Days

    3 Days

    sg_debug.log

    false

    2 Days

    1 Day
    Each log level and its parameters are described in the logging.$level property reference.

    Log File Rotation

    Log files are rotated once each exceeds a threshold max_size in megabytes. Once rotated, the log files are compressed (gzip) to reduce the disk usage. Aged logs are cleaned up once their age exceeds max_age days.

    Configure log rotation using the logging.level.$rotation property.

    For pre-2.1 log rotation — see: Log Rotation pre-2.1

    Log Redaction

    All log outputs can, optionally, be redacted, removing any user-data considered private.

    Enable this feature by setting the logging.redaction_level property.
